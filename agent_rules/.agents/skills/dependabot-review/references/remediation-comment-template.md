# Remediation Comment Templates

Templates for posting actionable comments on Hold and Caution Dependabot PRs.
Comments are posted via `gh pr comment {N} --body "..."`.

## Key Principles

- **Reuse agent data** — All analysis data is already gathered during the Review Phase. Never re-analyze.
- **Actionable checklist** — Every comment must include concrete next steps with commands.
- **Include gathered findings** — Embed the changelog summary, breaking changes assessment, and codebase impact from the agent's analysis.
- **Trackable marker** — Include `<!-- dependabot-review-bot -->` HTML comment for identification.

---

## Hold PR Template (⛔)

Use for PRs that cannot be merged due to CI failure, known breaking changes, or other blockers.

```markdown
<!-- dependabot-review-bot -->
## Dependabot Review: Automated Analysis

**Status**: ⛔ Hold — {primary_blocker_reason}
**Risk Level**: {High/Medium}
**Analyzed**: {date}

---

### Blocker

{1-2 sentence description of why this PR is blocked, e.g., "CI `test_and_compile` check failed. This is a pre-1.0 library with a 2-minor-version jump (0.11.2 → 0.13.1), increasing the likelihood of breaking changes."}

### Analysis Summary

**Version**: {old_version} → {new_version} ({change_type})
**Usage Scope**: {N} files ({list key packages/modules used})
**Security**: {Yes (CVE detail) / No}

#### Changelog Highlights ({old_version} → {new_version})

{Bullet list of key changes between versions, extracted from agent analysis.
Focus on: breaking changes, API modifications, deprecations, notable features.
Include version numbers for multi-version jumps.}

#### Breaking Changes Assessment

| Change | Affects Project? | Detail |
|--------|-----------------|--------|
| {breaking change 1} | {Yes/No/Unknown} | {explanation} |
| {breaking change 2} | {Yes/No/Unknown} | {explanation} |

#### Codebase Impact

{Summary of how the project uses this library — which packages, how many files,
what patterns. From agent's usage search results.}

**Key files**:
- `{path/to/file1.go}` — {brief usage description}
- `{path/to/file2.go}` — {brief usage description}

### Investigation Checklist

- [ ] **Check CI failure log**: `gh run view {run_id} --log-failed`
- [ ] **Classify failure**: compilation error / test failure / dependency conflict
- [ ] {Additional investigation steps specific to this PR...}
- [ ] **If dependency conflict**: try `@dependabot rebase`
- [ ] **If API breaking change**: identify affected code and plan migration

### Alternative Approaches

{If applicable — e.g., "Pin to v0.12.3 for bug fixes without breaking changes",
"Wait for upstream fix", "Create manual upgrade branch for incremental migration".
Omit this section if no meaningful alternative exists.}

### Compatibility

| Item | Value |
|------|-------|
| Library minimum | {language_version} |
| Project version | {project_version} |
| Compatible | {Yes/No} |

---
<sub>Generated by dependabot-review skill</sub>
```

## Caution PR Template (⚠️)

Use for PRs where CI passes but there are conditions to verify before merging
(e.g., runtime version upgrade, team decision needed, transitive dependency concerns).

```markdown
<!-- dependabot-review-bot -->
## Dependabot Review: Automated Analysis

**Status**: ⚠️ Caution — {primary_concern}
**Risk Level**: {Medium/Low}
**Analyzed**: {date}

---

### Concern

**Version**: {old_version} → {new_version} ({change_type})

**Security**: {Yes (CVE detail) / No}

#### Breaking Changes Assessment

{Summary from agent analysis.}
```

## Formatting Guidelines

### Variable Substitution

All `{variables}` must be filled from the agent's analysis results gathered in the Review Phase.

| Variable | Source |
|----------|--------|
| `{primary_blocker_reason}` | Agent's recommended action rationale |
| `{old_version}`, `{new_version}` | PR title parsing |
| `{change_type}` | Classification from Step 2 |
| `{date}` | Current date |
| `{N} files` | Agent's usage search count |
| `{run_id}` | Extract from `gh pr checks` output |
| Changelog / Breaking changes | Agent's changelog analysis |
| Codebase impact / Key files | Agent's usage search + file reading |

### Checklist Design Rules

1. **Start with the most impactful action** — the one most likely to unblock the PR
2. **Include concrete commands** where possible (`gh run view`, `go build ./...`, etc.)
3. **Keep to 3-6 items** — too many items reduce actionability
4. **Distinguish investigation vs. action items** — "Check X" vs. "Update Y"

### Length Guidelines

- **Hold comments**: 300-500 words (more detail needed due to blockers)
- **Caution comments**: 200-400 words (CI passed, focus on conditions)
- Trim verbose changelog entries — include only entries relevant to this project

### Execution Order

When posting multiple comments:
1. Post all comments in parallel using separate `gh pr comment` calls
2. Verify each comment was posted successfully
3. Report any failures in the result report
