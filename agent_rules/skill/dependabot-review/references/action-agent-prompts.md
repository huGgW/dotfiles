# Action Phase Agent Prompt Templates

Templates for agents that execute merges and post remediation comments in parallel.

---

## Comment Agent Prompt Template

Each Comment Agent posts remediation comments on 1-3 assigned ⚠️/⛔ PRs.
The analysis data is passed directly in the prompt — no re-analysis needed.

```
Post remediation comments on the following Dependabot PRs.
Use `gh pr comment {N} --body "..."` for each PR.

## Assigned PRs

{comment_assignments}

Each assignment below contains:
- PR number
- Comment body (ready to post)

### PR #{N1}

Post this comment:

```bash
gh pr comment {N1} --body "$(cat <<'COMMENT_EOF'
{pre_built_comment_body_1}
COMMENT_EOF
)"
```

### PR #{N2}

Post this comment:

```bash
gh pr comment {N2} --body "$(cat <<'COMMENT_EOF'
{pre_built_comment_body_2}
COMMENT_EOF
)"
```

## Instructions

1. Post each comment using the exact `gh pr comment` commands above.
2. If a comment post fails, retry once. If it fails again, record the error.
3. Do NOT modify the comment content.
4. Do NOT analyze or re-read any PRs.

## Output Format

For each PR, report:
- PR #{N}: ✅ Comment posted / ❌ Failed (reason)
```

### Template Variables

| Variable | Description |
|----------|-------------|
| `{comment_assignments}` | Summary of assignments for context |
| `{N1}`, `{N2}` | PR numbers |
| `{pre_built_comment_body_1}` | Complete comment body from review data + remediation template |

### How to Build Comment Bodies

The main agent constructs each comment body BEFORE launching the Comment Agent,
using the review data from Step 3 agents and the templates in `remediation-comment-template.md`.

This means: **comment content is fully prepared at launch time, not generated by the agent.**

## Independent Wave Merge Agent Prompt Template

Handles Wave 1 — truly independent PRs with no shared dependency file conflicts.

```
Merge the following independent Dependabot PRs via squash merge.

## Target PRs (no dependency relationship between them)

| PR# | Package | Version |
|-----|---------|---------|
{pr_rows}

## Strategy

1. Attempt to merge ALL PRs in parallel:
   Run `gh pr merge {N} --squash` for each PR.

2. Collect results. For any PR that fails with "not mergeable":
   a. Request rebase: `gh pr comment {N} --body "@dependabot rebase"`
   b. After requesting rebase for ALL failed PRs, wait 90 seconds.
   c. Check mergeable status: `gh pr view {N} --json mergeable --jq '.mergeable'`
   d. Retry merge for PRs that show MERGEABLE.

3. Maximum 3 retry rounds. If a PR still fails after 3 rounds, record it as failed.

## Important

- Only use `gh pr merge {N} --squash`. No other merge methods.
- Do NOT modify any code or create commits.
- Do NOT close PRs manually.
- If a PR shows "CONFLICTING" after rebase, request rebase again.

## Output Format

For each PR, report:

| PR# | Package | Status | Detail |
|-----|---------|--------|--------|
| {N} | {pkg}   | ✅ Merged / ❌ Failed | (error detail if failed) |

Also report:
- Total merged: N/M
- Total rounds needed: N
- Any PRs needing manual intervention
```

## Dependency Group Merge Agent Prompt Template

Handles Wave 2+ — interdependent PR groups that must be merged in a specific order.

```
Merge the following dependency group PRs in the specified order.
These PRs are interdependent — merge order matters.

## Merge Order (MUST follow this sequence)

| Order | PR# | Package | Version |
|-------|-----|---------|---------|
{ordered_pr_rows}

## Strategy

For each PR in order:

1. Attempt merge: `gh pr merge {N} --squash`

2. If merge succeeds:
   - **Check subsumed PRs**: If this PR subsumes other PRs (noted in the merge order),
     check their state: `gh pr view {SUBSUMED_N} --json state --jq '.state'`
     - If auto-closed → report as "🔄 Auto-closed (subsumed by #{N})"
     - If still open → close with comment:
       `gh pr close {SUBSUMED_N} --comment "Closed: changes already included in #{N}"`
   - Request rebase for ALL remaining (not-yet-merged, non-subsumed) PRs in the list:
     `gh pr comment {N} --body "@dependabot rebase"` for each remaining PR
   - Wait 90 seconds for rebases to complete.
   - Proceed to next PR.

3. If merge fails with "not mergeable":
   a. Request rebase: `gh pr comment {N} --body "@dependabot rebase"`
   b. Wait 90 seconds.
   c. Check: `gh pr view {N} --json mergeable --jq '.mergeable'`
   d. Retry merge (max 3 retries per PR).
   e. If still failing after 3 retries, skip this PR and continue with next.

4. After merging all possible PRs, report results.

## Important

- Follow the merge order strictly — core/base packages first, dependents last.
- Only use `gh pr merge {N} --squash`.
- Do NOT modify any code or create commits.
- After each successful merge, ALWAYS rebase all remaining PRs before attempting the next.
- This rebase step is critical: shared lock files (go.sum, etc.) cause cascading conflicts.

## Output Format

For each PR, report:

| Order | PR# | Package | Status | Rounds |
|-------|-----|---------|--------|--------|
| 1 | {N} | {pkg} | ✅ Merged / ❌ Failed / ⏭️ Skipped / 🔄 Auto-closed (subsumed) | N |

Also report:
- Total merged: N/M
- Total time estimate: ~N minutes
- Any PRs needing manual intervention
```

## Agent Assignment Guidelines

### Comment Agents

- Batch 2-3 comments per agent
- All comment agents launch simultaneously with `run_in_background: true`
- Comment bodies are pre-built by the main agent before launch
- **Comments are always posted on all ⚠️/⛔ PRs** — even when ⚠️ PRs are being merged.
  The comment serves as a traceable record of the risk assessment and conditions evaluated.

### Merge Agents

- **1 agent per wave** (not 1 per PR)
- Wave 1 (independent): single agent handles all independent PRs
- Wave 2+ (groups): 1 agent per dependency group
- All merge agents launch simultaneously with `run_in_background: true`
- Groups that don't share lock file entries CAN run in parallel

### Parallel Launch

All agents (comment + merge) launch in a **single message** with multiple Agent tool calls.
This maximizes parallelism:

```
Single message with N tool calls:
├── Agent: Comment Agent 1 (⚠️ PRs) — background
├── Agent: Comment Agent 2 (⛔ PRs) — background
├── Agent: Wave 1 Merge (independent PRs) — background
├── Agent: Wave 2 Merge (Dependency Group A, ordered) — background
└── Agent: Wave 3 Merge (Dependency Group B, ordered) — background
```

### Inter-Wave Dependencies

If Wave 2+ groups share lock file entries with Wave 1 PRs (common in Go projects),
the merge agents handle this through their built-in rebase-retry loops.
No explicit inter-agent coordination is needed — each agent independently resolves conflicts.

### Maximum Agent Count

- Comment Agents: ceil(non_merge_PRs / 3)
- Merge Agents: 1 + number_of_dependency_groups
- Total cap: 10 agents (same as review phase)
